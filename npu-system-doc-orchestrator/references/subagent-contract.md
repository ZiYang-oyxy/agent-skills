# Subagent 章节任务约定

用这个约定向 subagent 派发章节任务，避免范围漂移和重复写作。

## 交给每个 Subagent 的输入

至少提供下面信息：

- 文档目标：例如“面向研发团队的 NPU 芯片软硬件系统介绍”
- 章节名称与范围：只写本章节，不跨章定稿
- 已知事实：来自用户、仓库、已有文档、表格、图纸、PPT
- 明确排除项：例如不写训练路径、推理路径、性能方法论、部署、运维、问题定位
- 目标深度：简版、中版、详版
- 输出格式：Markdown

## 要求每个 Subagent 交付的内容

固定要求 subagent 输出 4 个部分：

1. **章节草稿**
   直接可并入主文档的 Markdown 内容。

2. **关键事实清单**
   列出该章节依赖的事实、数据、接口或架构点，方便主 agent 做交叉校对。

3. **未决问题或假设**
   资料不足时，不要编造；列出假设或缺口。

4. **建议图表**
   如果图或表能显著提高清晰度，提出建议；只有在确实需要时再画 Mermaid。

## 对 Subagent 的硬约束

- 不要发明芯片规格、带宽数字、算子支持结论、性能数据。
- 不要替其他章节下结论。
- 不要擅自扩展出训练路径、推理路径、benchmark 分析或能力边界章节。
- 不要扩展到部署、运维、告警、故障定位，除非明确在范围内。
- 资料不够时，交“假设”和“缺口”，不要强行补齐。
- 讨论总线、互联、内存时，要写清控制流、数据流、仲裁点和软件可见接口，不要只列模块名。

## 推荐派发模板

可直接复用下面的提示骨架：

```text
为一份“面向研发团队的 NPU 芯片软硬件系统介绍文档”撰写章节草稿。

你的章节范围：
<章节名称与边界>

已知事实：
<已知资料或要点>

明确不写：
<排除项>

输出要求：
1. 给出可直接并入主文档的 Markdown 章节草稿。
2. 单独列出关键事实清单。
3. 单独列出未决问题或假设。
4. 如确有必要，再提出建议图表，不要默认画图。

约束：
- 不要发明规格或数据。
- 不要重写其他章节。
- 保持工程化表述，少用宣传性措辞。
- 不要扩展到训练路径、推理路径、benchmark 或部署运维。
```

## 默认章节映射

长文档建议：

- Subagent 1: 方案概述 + 设计目标与范围
- Subagent 2: 软件总体架构
- Subagent 3: 芯片计算与控制架构
- Subagent 4: 总线、片上互联、外部接口与内存层次
- Subagent 5: 软硬件协同关键机制

中短文建议：

- Subagent A: 概述 + 软件总体架构
- Subagent B: 芯片架构
- Subagent C: 互联/内存 + 软硬件协同

## 主 Agent 保留职责

不要把下面内容下放给 subagent 定稿：

- 标题
- 摘要/概述
- 术语统一
- 章节间过渡
- 跨章节接口表
- 最终一致性与去重
